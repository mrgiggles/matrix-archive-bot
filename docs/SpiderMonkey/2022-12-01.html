<tcampbell>
The biggest restriction for being on a single thread at a time is that the garbage collected memory heap is not threadsafe (since there is a lot of bookkeeping needed that is constantly updated). Spidermonkey does allow creating alternate gc heaps for use by other threads, but these threads are then independent from eachother. Their communication goes through message passing using serialized data instead of bare objects. This is what underlies web Workers and postMessage.
16:00
<tcampbell>
In certain engine-internal cases some work can be done on a different thread from the JSContext as long as it doesn't involve the GC heap. This includes the ability to parse JS to an intermediate JS::Stencil format, and doing some JIT compile work (once the prep process that reads from the heap is done on the original thread and saved).
